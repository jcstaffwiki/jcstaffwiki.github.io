J.C.STAFF资料库 作品JSON填写说明

本文用于说明 data\works 里的作品JSON应该怎么填写。

本文中的“作品库”，指网页顶部导航里的“作品库”页面。
这个页面会把 data\works\index.json 里登记的作品读取出来，按年份和季度显示成一张张作品卡片。
data\works 文件夹是作品JSON文件存放的位置；网页上的“作品库”是读这些文件后生成的列表页面。

最重要的一句话：
凡是从官网、OP、ED、片尾、PV、访谈资料等来源抄来的名称表记，尤其是人名、公司名、团体名，除非已确认错误表记，否则请按原表记填写，不要翻译、不要简化、不要改汉字、不要自行补全。
职位名称按照单独的职位翻译映射表来写。
如果名称表记里有括号，例如“瀧川三智（REAL-T）”，直接照抄。网页会自动把括号内文字当作备注隔开显示，人物链接会指向括号外的名字“瀧川三智”。


一、可直接复制的基础模板

下面是一个基础模板。新建作品时，可以先复制它，再逐项改内容。
注意：JSON里不能写注释，真正放进 .json 文件时不要加入“说明文字”。

{
  "slug": "作品slug",
  "seasonYear": 2026,
  "season": "冬",
  "title": "作品中文名",
  "title_original": "作品原名",
  "type": "TV",
  "sourceCategory": "漫改",
  "episodes": 12,
  "premiere": "2026.01.01",
  "links": {
    "official": "https://example.com",
    "bangumi": "https://bgm.tv/subject/000000",
    "moegirl": "https://zh.moegirl.org.cn/作品名",
    "seesaa": "https://seesaawiki.jp/example",
    "sakugabooru": "https://www.sakugabooru.com/post?tags=example"
  },
  "cn_streaming": null,
  "copyright": "© COPYRIGHT",
  "production_info": {
    "main_staff_columns": [
      [
        ["监督", "监督名"],
        ["角色设计", "角色设计名"]
      ],
      [
        ["系列构成", "系列构成名"],
        ["动画制作", "J.C.STAFF"]
      ]
    ]
  },
  "detailed_staff": {
    "overview_headers": ["脚本", "分镜", "演出", "作画监督", "制作进行"],
    "episodes": [
      {
        "label": "第1话",
        "staff": [
          ["脚本", "脚本名"],
          ["分镜", "分镜名"],
          ["演出", "演出名"],
          ["作画监督", ["作画监督A", "作画监督B"]],
          ["制作进行", "制作进行名"]
        ]
      }
    ]
  }
}

更完整、覆盖各种特殊情况的范例，请看：
data\works\sample-work.json


二、JSON基础规则

1. JSON里的符号必须使用英文符号
包括：
英文双引号 " "
英文冒号 :
英文逗号 ,
英文中括号 [ ]
英文大括号 { }

不要使用中文引号“”、中文冒号：、中文逗号，、中文括号【】。

2. 项名不能随便改
例如：
"slug"
"seasonYear"
"title"
"production_info"
"detailed_staff"

这些项名是前端代码固定读取的，不能改成别的名字。

3. 文字要加双引号，数字不要加
正确：
"title": "某作品"
"episodes": 12

错误：
"title": 某作品
"episodes": "12"   （能显示，但不推荐）

4. 逗号规则
同一个大括号或中括号里，只要后面还有下一项，当前项末尾就要加逗号。
最后一项后面不要加逗号。

正确：
"title": "某作品",
"type": "TV"

错误：
"title": "某作品"
"type": "TV"

错误：
"title": "某作品",
"type": "TV",

5. 空内容怎么写
如果某项确实没有内容，可以删除这一项。
如果说明里要求没有内容时写 null，就写 null，不加引号。

正确：
"links": null

错误：
"links": "null"


三、文件位置与新增作品步骤

1. 每部作品一个 JSON 文件
作品文件放在：
data\works

例如：
data\works\现在的是哪一个多闻.json

2. slug 必须和文件名一致
如果文件名是：
现在的是哪一个多闻.json

那么 JSON 里必须写：
"slug": "现在的是哪一个多闻"

3. index.json 必须加入 slug
新增作品后，还要打开：
data\works\index.json

把新作品 slug 加进去，例如：
{
  "works": [
    "sample-work",
    "现在的是哪一个多闻",
    "新作品slug"
  ]
}

只有加入 index.json，作品库才会读取这部作品。

4. slug 命名建议
slug 不会直接显示在页面上，但会用于文件名、链接和封面匹配。
建议：
尽量短一点
最好用作品中文名或方便识别的名字
不要加空格
不要使用 / \ : * ? " < > | 这类文件名危险字符
同一部作品的 slug、JSON 文件名、封面文件名必须一致


四、封面图片规则

1. 封面放置位置
封面放在：
data\covers

2. 封面文件名
封面文件名必须和作品 slug 一致。

例如作品 slug 是：
"slug": "现在的是哪一个多闻"

封面可以命名为：
现在的是哪一个多闻.webp
现在的是哪一个多闻.jpg
现在的是哪一个多闻.png

3. 支持格式
支持：
.webp
.jpg
.png

推荐优先使用 .webp，因为体积更小，网页加载更快。

4. 图片比例
推荐竖版封面。
大致比例推荐 3:4 到 2:3 之间。
太横的图会被裁切或留空，看起来不适合作品卡片。

5. 图片大小
不需要放超高清原图。
建议单张封面控制在 200KB 到 500KB 左右。
如果图片特别重要、细节较多，也尽量不要超过 1MB。
以后作品越来越多，图片太大会明显拖慢网页。

6. 没有封面也能显示
如果没有找到封面，网页会显示渐变占位图，不会报错。
但正式作品建议尽量补封面。


五、基础信息字段说明

1. slug（必填）
作品的定位名。
必须和 JSON 文件名一致。
必须在 data\works\index.json 里登记。

示例：
"slug": "现在的是哪一个多闻"

2. seasonYear（必填）
作品在作品库里归入哪一年。
只写数字，不加引号。

示例：
"seasonYear": 2026

这项不一定必须机械等于 premiere 的年份。
如果你判断某作品应归入 2026 冬番，就填 2026。

3. season（必填）
作品在作品库里归入哪个季度。
只能填写：
"冬"
"春"
"夏"
"秋"

一般参考：
1月开播：冬
4月开播：春
7月开播：夏
10月开播：秋

如果遇到特殊播出时间，可以按本资料库自己的分类判断。

4. title（必填）
作品中文名。
显示在作品详情页大标题和作品卡片上。

中文名建议优先级：
中国大陆正版译名
中国大陆官方宣传译名
繁中官译转简体
原作小说、漫画、游戏等官方译名
常用社区译名

5. title_original（建议填写）
作品原名。
显示在详情页中文标题下方。
这里不是专指日本名，而是该动画原发行地区、原始官方使用的标题。

示例：
日本动画通常填写日文原名。
如果以后收录的作品原发行地区不是日本，就填写该地区官方原名。

请使用官方动画标题，不要自行翻译。

6. type（必填）
作品形式。
只能填写：
"TV"
"剧场版"
"OVA"
"OAD"
"WEB"
"MV"
"制作协力"
"其他"

说明：
如果 J.C.STAFF 是主制作，按作品本身形式填 TV、剧场版等。
如果 J.C.STAFF 负责某一整集、OP、ED、PV、CM、LIVE part 等完整段落的制作协力，且本站决定记录这一条，填 "制作协力"。
如果只是部分外包，例如一部分原画、第二原画、动画、仕上、背景、摄影等工作外包给 J.C.STAFF，不能算作本站作品条目的 "制作协力"。
简单说：必须是某一整段或某一整集交由 J.C.STAFF 制作协力，才作为 "制作协力" 条目记录。
"其他" 用于 CM、活动动画等不适合归入前面类型的情况。

7. sourceCategory（必填）
来源类型。
只能填写：
"轻改"
"漫改"
"游戏改"
"原创"
"其他"

8. episodes（通常填写）
作品集数。

已完结：
"episodes": 12

未完结：
"episodes": "更新中"

未开播：
"episodes": "未开播"

如果 type 是 "制作协力"，通常不填 episodes，改填 responsibility。

9. responsibility（制作协力时填写）
当 type 是 "制作协力" 时，用它写 J.C.STAFF 负责的内容。

示例：
"responsibility": "第7/8话・OP"

普通主制作作品通常不需要这一项。

10. premiere（建议填写）
首播日期。
显示在详情页基本信息里。
格式建议：
"2026.01.01"


请尽量按北京时间 24 小时制判断首播日期。

11. links
作品相关资料链接。
显示在详情页“相关链接”。

基础款固定键名有五个：
"official"：官方网站
"bangumi"：Bangumi
"moegirl"：萌娘百科
"seesaa"：Seesaawiki
"sakugabooru"：SAKUGABOORU

这五个链接在网页上的显示顺序是固定的：
官方网站
Bangumi
萌娘百科
Seesaawiki
SAKUGABOORU

填写时也建议按这个顺序往上填。
有哪个填哪个，缺哪个就删掉那一行。
只有这五个基础链接全都没有时，才考虑写：
"links": null

示例：
"links": {
  "official": "https://example.com",
  "bangumi": "https://bgm.tv/subject/000000",
  "moegirl": "https://zh.moegirl.org.cn/作品名",
  "seesaa": "https://seesaawiki.jp/example",
  "sakugabooru": "https://www.sakugabooru.com/post?tags=example"
}

如果还想加入其他链接，也可以继续往后写。
额外链接会显示在五个基础链接后面。
额外链接的“键名”，就是 links 里面冒号左边的那段文字。
例如下面这行里，annict 就是键名：
"annict": "https://annict.com/works/0000"

固定的五个键名会被网页自动显示成中文或常用名。
额外链接没有自动翻译，网页会直接把键名显示成链接名。
所以额外链接的键名建议写得短一点、能看懂一点。

示例：
"links": {
  "official": "https://example.com",
  "bangumi": "https://bgm.tv/subject/000000",
  "moegirl": "https://zh.moegirl.org.cn/作品名",
  "seesaa": "https://seesaawiki.jp/example",
  "sakugabooru": "https://www.sakugabooru.com/post?tags=example",
  "annict": "https://annict.com/works/0000"
}

链接必须以 http:// 或 https:// 开头。
推荐统一使用 https://。

12. cn_streaming
中国大陆地区正版流媒体链接。
显示在详情页“中国大陆正版”。

有正版平台时：
"cn_streaming": [
  { "title": "哔哩哔哩", "url": "https://www.bilibili.com/bangumi/play/xxxx" },
  { "title": "爱奇艺", "url": "https://www.iqiyi.com/xxxx.html" }
]

没有中国大陆正版流媒体时：
"cn_streaming": null

只填写中国大陆地区正版，不填港澳台、日本、欧美等地区平台。

13. copyright
版权标。
显示在详情页基本信息底部。
请按官方版权标原样填写。

示例：
"copyright": "©師走ゆき・白泉社／多聞くん今どっち！？製作委員会"


六、表记填写总原则

这一节非常重要。

1. 名称表记不要改
凡是来自官网、OP、ED、片尾等来源的人名、公司名、团体名，都按原表记填写。
职位名称请通过职位翻译映射表统一处理。

不要做这些事：
不要把人名里的繁体字、旧字体、新字体自行改掉
不要把公司名、团体名翻译成自己习惯的中文
不要把英文公司名、团体名的大小写改掉
不要把公司名、团体名简写成自己习惯的写法
不要自行合并或拆分名称表记

例如名称原表记是：
瀧川三智（REAL-T）

就写：
["编集", "瀧川三智（REAL-T）"]

不要改成：
["编集", "泷川三智（REAL-T）"]

2. 括号备注直接照抄
例如：
["编集", "瀧川三智（REAL-T）"]

网页会显示为：
瀧川三智（REAL-T）

其中“瀧川三智”会成为人物链接，括号里的“REAL-T”会作为备注显示。

3. 总览归类不等于修改分集显示
详细制作人员里有时需要把一个职位归入总览表某一列。
例如分集里写的是：
LIVE part制作进行

但总览表希望它出现在“制作进行”列。
此时可以写：
["LIVE part制作进行", "江川風太", "制作进行"]

这里第1项“LIVE part制作进行”会在分集详情里显示。
第3项“制作进行”只用于总览表归类，不会改变人名、公司名等名称表记。


七、主要制作人员 production_info

主要制作人员显示在作品详情页的“主要制作人员”区域。
适合填写官网、PV、宣传媒介、OP/ED固定表记，以及从单集表记中总结出来的统一 staff。

基本结构：
"production_info": {
  "main_staff_columns": [
    [
      ["左列职位", "左列人名"]
    ],
    [
      ["右列职位", "右列人名"]
    ]
  ]
}

main_staff_columns 必须是左右两列。
第一个 [ ] 是左列。
第二个 [ ] 是右列。

1. 单人写法
["监督", "永岡智佳"]

2. 多人写法
["总作画监督", ["千葉充", "露木愛里", "白浜有"]]

3. 公司也按人名位置写
["动画制作", "J.C.STAFF"]

4. 空行写法
空行不只是为了好看，也用于区分资料来源。

通常建议：
空行以上：官网、PV、宣传媒介、OP/ED固定表记等明确统一公布的主创。
空行以下：根据每集表记总结出来的统一 staff，例如动画制作制片人、设定制作、摄影管理等。

如果希望某一列中间空一行，可以写：
"空行"

也可以写：
"-"

推荐用 "空行"，更容易看懂。

八、同名人物映射

默认情况下，网页会把人名本身当作人物页文件名。

例如：
["监督", "永岡智佳"]

点击后会去找：
data\people\永岡智佳.json

如果没有这个人物文件，也不会报错，只是人物页暂无信息。

但如果存在同名同姓不同人，就要做映射。

1. 单个同名
例如“山田太郎”在这里指演出，不是作监：

["演出", "山田太郎", { "山田太郎": "山田太郎(演出)" }]

网页显示仍然是“山田太郎”。
但点击时会去找：
data\people\山田太郎(演出).json

2. 多人里只有一个人需要映射
例如只有山田太郎需要区分，王五、赵六不需要：

["制作进行", ["山田太郎", "王五", "赵六"], { "山田太郎": "山田太郎(制作)" }]

3. 同一个职位里出现两个完全同名的人
这种情况不能用上面的名字映射，因为两个名字一样，无法判断哪个是哪个。
要用“按位置映射”的写法。

["作画监督", ["山田太郎", "山田太郎"], ["山田太郎(作监A)", "山田太郎(作监B)"]]

第一个山田太郎对应第一个映射。
第二个山田太郎对应第二个映射。

4. 同组同名还混着其他人
例如：
山田太郎、hei、山田太郎

其中两个山田太郎要区分，hei 不需要映射。
此时映射数组必须和人名数组数量一样。
不需要映射的位置写空字符串 ""。

["第二原画", ["山田太郎", "hei", "山田太郎"], ["山田太郎(二原A)", "", "山田太郎(二原B)"]]

效果：
第一个山田太郎 -> 山田太郎(二原A)
hei -> hei
第二个山田太郎 -> 山田太郎(二原B)


九、详细制作人员 detailed_staff

详细制作人员显示在作品详情页下方。
它有两个显示方式：
总览表：按集数横向整理几个重点职位
分集详情：点击 OP、第1话、第2话等标签后，看该集完整 staff 列表

基本结构：
"detailed_staff": {
  "overview_headers": ["脚本", "分镜", "演出", "作画监督", "制作进行"],
  "episodes": [
    {
      "label": "第1话",
      "staff": [
        ["脚本", "永井千晶"],
        ["分镜", "永岡智佳"],
        ["演出", "丸山由太"]
      ]
    }
  ]
}

1. overview_headers
总览表表头。
这里写你希望总览表横向显示哪些职位。

示例：
"overview_headers": ["脚本", "分镜", "演出", "总作画监督", "制作进行", "制作协力"]

注意：
overview_headers 是本资料库为了表格整理用的列名。
它不等于要求你修改名称表记。
每条 staff 的第1项是职位显示名，后续会配合职位翻译映射表统一处理。
每条 staff 的第2项人名、公司名、团体名，必须按原表记写。

2. episodes
逐集列表。
每一项代表一个 OP、ED、某一话、LIVE part、OVA 等。

示例：
{
  "label": "第1话",
  "staff": [
    ["脚本", "永井千晶"],
    ["分镜・演出", "永岡智佳"],
    ["总作画监督", "伊東葉子"],
    ["制作进行", "永井健太"]
  ]
}

3. label
该段 staff 的名称。
可以写：
"OP"
"ED"
"第1话"
"第12话"
"LIVE part动画（第5话）"

4. staff
该集/该段的制作人员。
写法和主要制作人员一样：

单人：
["脚本", "永井千晶"]

多人：
["总作画监督", ["露木愛里", "千葉充", "白浜有"]]

有括号备注：
["作画管理制作", "磯田侑（米川）"]

同名映射：
["分镜", "山田太郎", { "山田太郎": "山田太郎(分镜)" }]

同组同名：
["作画监督", ["山田太郎", "山田太郎"], ["山田太郎(作监A)", "山田太郎(作监B)"]]

5. 总览归类
有些职位显示名较长，但希望在总览表里归入某个表头。

写法：
["职位显示名", "名称原表记", "总览表列名"]

示例：
["LIVE part制作进行", "江川風太", "制作进行"]

含义：
分集详情里显示“LIVE part制作进行”
总览表里归入“制作进行”这一列

6. 同时需要总览归类和人名映射
如果既要把职位归入总览列，又要处理同名人物，用四项写法：

["特殊分镜", "山田太郎", "分镜", { "山田太郎": "山田太郎(特殊分镜)" }]

如果是同组同名：
["特殊作画监督", ["山田太郎", "hei", "山田太郎"], "作画监督", ["山田太郎(特作监A)", "", "山田太郎(特作监B)"]]

四项含义：
第1项：职位显示名
第2项：人名、公司名、团体名，或这些名称组成的数组
第3项：归入总览表哪一列
第4项：人名映射

7. 不会进入总览表的职位
如果某个职位不在 overview_headers 里，也没有写第3项归类，它不会出现在总览表。
但它仍然会出现在分集详情里。

例如：
["原画", ["山田太郎", "佐藤花子"]]

如果 overview_headers 没有“原画”，总览表不显示它。
分集详情仍然显示它。


十、staff数组写法总表

普通单人：
["职位显示名", "名称原表记"]

普通多人：
["职位显示名", ["名称A", "名称B"]]

单人同名映射：
["职位显示名", "名称原表记", { "名称原表记": "人物文件slug" }]

多人中某些人同名映射：
["职位显示名", ["名称A", "名称B"], { "名称A": "人物文件slug" }]

同组同名按位置映射：
["职位显示名", ["同名A", "同名A"], ["人物文件slug1", "人物文件slug2"]]

同组同名混合其他人：
["职位显示名", ["同名A", "普通名称", "同名A"], ["人物文件slug1", "", "人物文件slug2"]]

总览归类：
["职位显示名", "名称原表记", "总览表列名"]

总览归类 + 单人同名映射：
["职位显示名", "名称原表记", "总览表列名", { "名称原表记": "人物文件slug" }]

总览归类 + 同组同名映射：
["职位显示名", ["同名A", "普通名称", "同名A"], "总览表列名", ["人物文件slug1", "", "人物文件slug2"]]

空行：
"空行"


十一、数据检查脚本

项目里有一个中文检查脚本：
tools\check-data.js

运行方式：
node tools\check-data.js

它会检查：
JSON 格式是否合法
index.json 里的作品文件是否存在
作品内部 slug 是否和文件名一致
season、type、sourceCategory 是否写在允许范围内
links 和 cn_streaming 链接是否像正常 http/https 链接
staff 是否符合当前数组写法
同组同名映射数组数量是否和人名数量一致
封面是否存在

检查结果里：
“错误”需要修。
“提醒”不一定会阻止页面显示，但建议看一下。

例如 sample-work 没有封面时，会提醒：
sample-work：没有找到封面图片，页面会显示渐变占位图

这个不影响范例正常读取。


十二、常见错误

1. 忘记在 index.json 加 slug
结果：作品文件存在，但作品库不显示。

2. slug 和文件名不一致
结果：详情页、封面、链接可能对不上。

3. 逗号多写或漏写
结果：整个 JSON 读取失败。

4. 把 null 写成 "null"
结果：网页会把它当成文字，不是空值。

5. 同组同名映射数量不一致
错误示例：
["第二原画", ["山田太郎", "hei", "山田太郎"], ["山田太郎A", "山田太郎B"]]

正确示例：
["第二原画", ["山田太郎", "hei", "山田太郎"], ["山田太郎A", "", "山田太郎B"]]

6. 把名称原表记改成自己的写法
不推荐：
["编集", "泷川三智（REAL-T）"]

推荐：
["编集", "瀧川三智（REAL-T）"]

十三、推荐填写顺序

新建作品时，建议按这个顺序来：

1. 复制一个已有 JSON，或复制本文最上方模板
2. 修改 slug、title、title_original、type、sourceCategory 等基础信息
3. 在 data\works\index.json 里加入 slug
4. 放入封面图片，并让封面文件名和 slug 一致
5. 填 production_info.main_staff_columns
6. 填 detailed_staff.overview_headers
7. 按 OP、ED、第1话、第2话顺序填写 detailed_staff.episodes
8. 遇到同名人物时再补人名映射
9. 运行 node tools\check-data.js
10. 打开网页确认显示效果

如果只想先让作品显示出来，可以先填基础信息和 production_info。
detailed_staff 可以之后慢慢补。
